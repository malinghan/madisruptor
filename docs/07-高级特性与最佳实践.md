# 第七章：高级特性与最佳实践

## 7.1 性能调优技巧

### 7.1.1 选择合适的 RingBuffer 大小

```
经验法则:
  1. 必须是 2 的幂: 256, 512, 1024, 2048, 4096, ...
  2. 太小 → 生产者频繁阻塞等待消费者
  3. 太大 → 浪费内存，缓存效率下降
  
推荐:
  - 低延迟场景: 1024 ~ 4096
  - 高吞吐场景: 4096 ~ 65536
  - 内存受限:   256 ~ 1024
  
计算方法:
  bufferSize ≈ 生产速率 × 最大可接受延迟
  例: 100万/秒 × 0.001秒 = 1000 → 选 1024
```

### 7.1.2 单生产者 vs 多生产者

```java
// 单生产者（性能更高，推荐）
Disruptor<Event> disruptor = new Disruptor<>(
    factory, bufferSize, threadFactory,
    ProducerType.SINGLE,  // ← 明确指定
    waitStrategy);

// 多生产者
Disruptor<Event> disruptor = new Disruptor<>(
    factory, bufferSize, threadFactory,
    ProducerType.MULTI,   // ← 有 CAS 开销
    waitStrategy);
```

**最佳实践**：即使有多个来源，也尽量汇聚到一个线程后再发布到 Disruptor（变成单生产者）。

### 7.1.3 使用 EventTranslator

```java
// 不推荐: 两步操作，可能忘记 publish
long seq = ringBuffer.next();
try {
    Event e = ringBuffer.get(seq);
    e.setValue(data);
} finally {
    ringBuffer.publish(seq);
}

// 推荐: 使用 EventTranslator，一步完成
private static final EventTranslatorOneArg<Event, Long> TRANSLATOR =
    (event, sequence, data) -> event.setValue(data);

ringBuffer.publishEvent(TRANSLATOR, data);
// 优势: 不会忘记 publish，更安全
```

### 7.1.4 Event 对象的清理

```java
// 推荐: 在消费完成后清理 Event，避免内存泄漏
public class ClearingHandler implements EventHandler<OrderEvent> {
    @Override
    public void onEvent(OrderEvent event, long sequence, boolean endOfBatch) {
        event.clear();  // 清空大对象引用
    }
}

// 将 ClearingHandler 放在消费链的最后
disruptor.handleEventsWith(businessHandler).then(clearingHandler);
```

## 7.2 常见陷阱

### 陷阱 1: EventHandler 中的异常

```java
// 错误: 异常会导致消费者停止！
public void onEvent(Event event, long seq, boolean endOfBatch) {
    process(event);  // 如果抛异常，默认会记录日志并继续
}

// 正确: 设置异常处理策略
disruptor.setDefaultExceptionHandler(new ExceptionHandler<Event>() {
    public void handleEventException(Throwable ex, long seq, Event event) {
        log.error("处理失败", ex);
        // 根据业务决定: 重试 / 跳过 / 告警
    }
    // ...
});
```

### 陷阱 2: Event 对象的可变性

```java
// 错误: 在 EventHandler 中保存 Event 的引用
public void onEvent(OrderEvent event, long seq, boolean endOfBatch) {
    pendingOrders.add(event);  // 危险! event 对象会被复用覆盖
}

// 正确: 复制需要保留的数据
public void onEvent(OrderEvent event, long seq, boolean endOfBatch) {
    OrderDTO dto = new OrderDTO(event.getOrderId(), event.getPrice());
    pendingOrders.add(dto);  // 安全
}
```

### 陷阱 3: 生产者忘记 publish

```java
// 错误: 异常时忘记 publish，导致 sequence 断裂
long seq = ringBuffer.next();
Event e = ringBuffer.get(seq);
e.setValue(process(data));  // 如果这里抛异常...
ringBuffer.publish(seq);    // 这行不会执行！

// 正确: 使用 try-finally
long seq = ringBuffer.next();
try {
    Event e = ringBuffer.get(seq);
    e.setValue(process(data));
} finally {
    ringBuffer.publish(seq);  // 确保一定发布
}
```

### 陷阱 4: 消费者处理时间过长

```java
// 问题: 慢消费者会阻塞生产者
public void onEvent(Event event, long seq, boolean endOfBatch) {
    Thread.sleep(1000);  // 模拟慢处理
    // 生产者 next() 会阻塞等待这个消费者
}

// 解决方案:
// 1. 使用 WorkerPool 并行消费
// 2. 增大 RingBuffer
// 3. 异步化耗时操作
// 4. 使用多级 Disruptor
```

### 7.3.2 多级 Disruptor 架构

```
适用于复杂的事件处理流水线:

Disruptor1 (接入层)          Disruptor2 (处理层)         Disruptor3 (持久层)
[RingBuffer1]               [RingBuffer2]              [RingBuffer3]
    ↓                           ↓                          ↓
[解码Handler] ──publish──► [业务Handler] ──publish──► [持久化Handler]
                                                          ↓
                                                       Database
```
